## 1. 系统架构设计

### 1.1 三层架构

YCL 解决方案 `YCL.sln` 由三个项目组成，按依赖方向自上而下分层：

```
┌─────────────────────────────────────┐
│  YCL（WPF 主程序，net10.0-windows）   │  ← View + ViewModel + 应用服务
└──────────────┬──────────────────────┘
               │ 引用
┌──────────────▼──────────────────────┐
│  YCL.Core（核心库，net10.0）         │  ← 业务逻辑、下载、启动、账户、模组
└──────────────┬──────────────────────┘
               │ 引用
┌──────────────▼──────────────────────┐
│  YCL.Models（数据模型，net10.0）     │  ← POCO 数据类、配置、枚举
└─────────────────────────────────────┘
```

#### 各层职责

| 层 | 项目 | 目标框架 | 职责 |
|----|------|----------|------|
| 表现层 | `YCL` | `net10.0-windows10.0.18362.0` | XAML 视图、ViewModel、应用级服务（导航/主题/配置）、主窗口、DI 容器装配 |
| 业务层 | `YCL.Core` | `net10.0` | 启动核心、下载系统、版本管理、账户认证、模组加载器、模组/资源管理、存档/截图、Java、更新检查、日志 |
| 模型层 | `YCL.Models` | `net10.0` | 纯数据类（`AppConfig`、`AccountBase` 及子类、`VersionInfo` 等），无业务逻辑 |

#### 分层约束

- **依赖方向单向**：`YCL → YCL.Core → YCL.Models`。`YCL.Models` 不依赖任何上层，`YCL.Core` 只依赖 `YCL.Models`。
- **避免循环引用**：`YCL.Core` 需要刷新令牌时不能反向引用 `YCL.Services`，因此 `AccountManager` 在构造时把 `RefreshMicrosoftAccountAsync` / `RefreshYggdrasilAccountAsync` 作为委托注入到 `MicrosoftAccount.RefreshHandler` / `YggdrasilAccount.RefreshHandler` 静态字段（见 `YCL.Core\Accounts\AccountManager.cs:62-64`）。
- **配置即时感知**：`YCL.Core` 不持有 `IConfigService`（避免反向依赖），而是通过 `Func<string>` / `Func<bool>` 委托读取最新配置。例如 `VersionManager` 接收 `Func<string> minecraftPathProvider`（见 `YCL.Core\Versions\VersionManager.cs:36-39`），`JavaDetector` 接收 `Func<string>` 类似处理。
- **Models 不依赖 WPF**：`AppConfig.AccentColorHex` 用 `string?` 而不是 `Color`，避免 `YCL.Models` 引入 `PresentationCore`。

### 1.2 MVVM 模式

YCL 严格遵循 **Model-View-ViewModel** 模式：

```
View (XAML)  ←─DataTemplate─→  ViewModel  ←─service─→  Model / Core Service
   │                              │
   └─ Binding（双向/单向）         └─ ObservableCollection 通知 UI 刷新
   └─ Command（按钮点击触发）
```

#### View（视图）

位置：`YCL\Views\*.xaml`

每个页面是一对文件：`XxxPage.xaml`（界面）+ `XxxPage.xaml.cs`（code-behind，通常为空或仅做轻量初始化）。所有页面共享 `MainWindow.xaml` 作为外壳。

XAML 通过 `App.xaml` 中的 `DataTemplate` 把 ViewModel 类型映射到对应 View：

```xml
<DataTemplate DataType="{x:Type vm:HomePageViewModel}">
    <views:HomePage/>
</DataTemplate>
```

`MainWindow.xaml` 的 `ContentControl` 绑定到 `MainWindowViewModel.CurrentView`，由 DataTemplate 自动渲染对应页面。

#### ViewModel

位置：`YCL\ViewModels\*.cs`

- 继承自 `ViewModelBase`（提供 `SetProperty<T>` 帮助方法与 `INotifyPropertyChanged` 实现）。
- 使用 `CommunityToolkit.Mvvm` 的 `[ObservableProperty]` 与 `[RelayCommand]` 源生成器自动生成属性与命令，减少样板代码。
- 通过 DI 注入业务服务（`IVersionManager`、`IGameLauncher` 等），不直接 new 服务实例。
- 启动页和下载页 ViewModel 注册为 **Singleton**（事件订阅生命周期一致）；存档页和截图页注册为 **Transient**（每次导航重建，确保数据最新）。

#### Model

位置：`YCL.Models\` + `YCL.Core\` 中的领域类

- `YCL.Models` 中的类是 POCO（Plain Old CLR Object），只承载数据。
- `YCL.Core` 中的服务类是业务逻辑载体，无 UI 依赖，可被任意 .NET 项目复用。

### 1.3 依赖注入容器

YCL 使用 `Microsoft.Extensions.Hosting` 的 `Host.CreateDefaultBuilder()` 构建通用主机，并在 `ConfigureServices` 中注册所有服务。

入口在 `YCL\App.xaml.cs`：

```csharp
_host = Host.CreateDefaultBuilder()
    .ConfigureServices((context, services) =>
    {
        // 1. 配置/主题/导航
        services.AddSingleton<IConfigService, ConfigService>();
        services.AddSingleton<IThemeService, ThemeService>();
        services.AddSingleton<INavigationService, NavigationService>();

        // 2. 核心业务服务（共 30+ 个注册）
        services.AddSingleton<IVersionResolver, VersionResolver>();
        services.AddSingleton<ILaunchArgumentBuilder, LaunchArgumentBuilder>();
        services.AddSingleton<IDownloadSource>(sp => { /* 工厂模式 */ });
        // ...

        // 3. ViewModel
        services.AddSingleton<MainWindowViewModel>();
        services.AddSingleton<LaunchPageViewModel>();
        // ...

        // 4. 主窗口
        services.AddSingleton<MainWindow>();
    })
    .Build();
```

#### 生命周期策略

| 生命周期 | 用途 | 示例 |
|----------|------|------|
| Singleton | 无状态服务 / 全应用共享状态 | `ConfigService`、`VersionManager`、`GameLauncher`、`MainWindow` |
| Transient | 每次解析都新建，需要最新数据 | `SettingsPageViewModel`、`AboutPageViewModel`、`SavesPageViewModel`、`ScreenshotsPageViewModel` |

#### 工厂模式注册

需要从配置读取参数的服务用工厂模式注册，例如：

```csharp
services.AddSingleton<IDownloadSource>(sp =>
{
    var config = sp.GetRequiredService<IConfigService>();
    return new DownloadSourceManager(config.Current.DownloadSource);
});
```

#### 收集多实现

`IModLoaderInstaller` 有 5 个实现（Fabric/Forge/Quilt/NeoForge/LiteLoader），全部注册为 `IModLoaderInstaller`，`ModLoaderManager` 通过 `IEnumerable<IModLoaderInstaller>` 在构造时收集所有实现。

### 1.4 异步编程模型

YCL 全面采用 `async/await` 异步编程：

- **所有 I/O 操作异步化**：HTTP 下载、文件读写、进程启动等全部用 `XxxAsync` 方法，避免阻塞 UI 线程。
- **取消令牌**：长任务（下载、登录、安装）接受 `CancellationToken`，允许用户中途取消。
- **进度报告**：用 `IProgress<T>` / `event EventHandler<T>` 反馈进度给 UI。
- **UI 线程切换**：后台线程更新 UI 用 `Dispatcher.BeginInvoke` 或 `SynchronizationContext`。
- **全局异常兜底**：`App.xaml.cs` 挂接三个异常入口：
  1. `DispatcherUnhandledException`（UI 线程，可阻止崩溃）
  2. `AppDomain.CurrentDomain.UnhandledException`（非 UI 线程，无法阻止）
  3. `TaskScheduler.UnobservedTaskException`（Task 未观察异常，`SetObserved` 阻止崩溃）

## 3. 实现细节

### 3.1 版本 JSON 解析与 inheritsFrom 递归合并

**位置**：`YCL.Core\Versions\VersionResolver.cs`

Minecraft 版本 JSON 可能含 `inheritsFrom` 字段，指向父版本（典型场景：Forge 版本继承原版）。YCL 用递归方式合并：

```
ReadAndMerge(minecraftPath, versionId):
  1. 读取 .minecraft/versions/<versionId>/<versionId>.json
  2. 反序列化为 VersionInfo（注册 ArgumentItemConverter 处理混合类型数组）
  3. 如果 info.InheritsFrom 非空：
     a. 递归 ReadAndMerge(minecraftPath, info.InheritsFrom)
     b. 合并 libraries（子版本优先，父版本补齐）
     c. 合并 arguments.jvm / arguments.game（父在前，子在后追加）
     d. 合并 mainClass（子覆盖父）
     e. 合并 releaseTime/type（子覆盖父）
  4. 返回最终合并后的 VersionInfo
```

`ArgumentItemConverter` 处理 `arguments.game` 数组中混合的 `string` 与 `{ "rules": [...], "value": ... }` 对象。

库路径解析规则（`ResolveLibraries`）：
- 按 `name` 字段（`group:artifact:version`）推路径：`libraries/<group>/<artifact>/<version>/<artifact>-<version>.jar`
- 按 `rules` 过滤（如 `os=windows` 才下载）
- 按 `natives` 取 classifier（如 `natives.windows` → `<artifact>-<version>-natives-windows.jar`）

### 3.2 多线程下载器实现

**位置**：`YCL.Core\Download\MultiThreadDownloader.cs`

**核心思路**：用 HTTP Range 请求把大文件切成 N 块并行下载。

**工作流程**：

1. **HEAD 请求**：获取文件总大小（`Content-Length`）与是否支持 Range（`Accept-Ranges: bytes`）。
2. **回退判断**：文件 < 1MB 或不支持 Range → 用 `FileDownloader` 单线程下载。
3. **分片**：按 `_threadCount`（默认 8，来自 `AppConfig.DownloadThreads`）把文件切成等大分片。
4. **并行下载**：每个分片独立发起 GET 请求（带 `Range: bytes=start-end` header），用 `FileStream.Seek` 写入 `.part` 文件对应位置。
5. **断点续传**：用 `.meta` 文件记录每个分片已下载字节数，下次启动时跳过已下载部分。
6. **进度汇总**：所有分片进度加总后通过 `ProgressChanged` 事件上报（节流 200ms 一次）。
7. **完成处理**：所有分片下载完成 → 删除 `.meta` → `.part` 重命名为最终文件名。
8. **重试机制**：单分片失败按 `_retryCount`（默认 3）重试。

**关键常量**：

```csharp
private const long MultiThreadThreshold = 1 * 1024 * 1024;  // 1MB 触发多线程
private const int BufferSize = 64 * 1024;                    // 64KB 缓冲
private const int ProgressIntervalMs = 200;                 // 进度上报间隔
```

### 3.3 微软账户 OAuth 设备代码流

**位置**：`YCL.Core\Accounts\MicrosoftAuthenticator.cs`

OAuth 2.0 设备代码流（Device Code Flow）6 阶段流程：

```
阶段 1：请求设备 code
  POST https://login.microsoftonline.com/consumers/oauth2/v2.0/devicecode
  → 拿到 user_code + device_code + verification_uri
  → 提示用户：浏览器打开 verification_uri 输入 user_code

阶段 2：轮询 token（用户在浏览器登录时）
  POST https://login.microsoftonline.com/consumers/oauth2/v2.0/token
  body: grant_type=urn:ietf:params:oauth:grant-type:device_code
  → 用户未完成时返回 authorization_pending，每 5 秒轮询一次
  → 用户完成后拿到 access_token + refresh_token

阶段 3：Xbox Live 认证
  POST https://user.auth.xboxlive.com/user/authenticate
  body: { Properties: { AuthMethod: "RPS", AuthToken: <access_token> }, RelyingParty: "..." }
  → 拿到 XBL token + userhash

阶段 4：XSTS 认证
  POST https://xsts.auth.xboxlive.com/xsts/authorize
  body: { Properties: { SandboxId: "RETAIL", UserTokens: [<XBL token>] }, RelyingParty: "..." }
  → 拿到 XSTS token + userhash

阶段 5：Minecraft 令牌
  POST https://api.minecraftservices.com/authentication/login_with_xbox
  body: { identityToken: "XBL3.0 x=<userhash>;<XSTS token>" }
  → 拿到 Minecraft access_token

阶段 6：获取 profile
  GET https://api.minecraftservices.com/minecraft/profile
  Authorization: Bearer <Minecraft access_token>
  → 拿到 username + uuid + 皮肤 URL
```

**关键参数**：
- `ClientId = "00000000402b5328"`（PCL 公开 ID，可通用）
- `Scope = "XboxLive.signin offline_access"`（`offline_access` 拿 refresh_token）
- HTTP 超时 30 秒
- 每阶段通过 `IProgress<MicrosoftLoginProgress>` 上报进度

### 3.4 DPAPI 令牌加密存储

**位置**：`YCL.Core\Accounts\AccountManager.cs`

**设计思路**：账户文件 `accounts.json` 中只存"非敏感"信息（用户名、UUID、类型、上次刷新时间），敏感令牌（access_token、refresh_token、XBL/XSTS token）用 DPAPI 加密后存到 `EncryptedTokens` 字段。

**保存流程**：

```
1. 收集每个账户的敏感字段：account.GetSensitiveTokens() → Dictionary<string, string>
2. JSON 序列化为字符串
3. UTF-8 编码为字节
4. DPAPI 加密（CurrentUser 范围，DataProtectionScope.CurrentUser）
   ProtectedData.Protect(bytes, null, DataProtectionScope.CurrentUser)
5. Base64 编码
6. 存入 account.EncryptedTokens
7. 整个账户列表 JSON 序列化写入 accounts.json
```

**加载流程**：

```
1. 读取 accounts.json，反序列化为 List<AccountBase>
   （多态 $type 字段决定具体子类）
2. 对每个账户：
   a. 读取 EncryptedTokens
   b. Base64 解码为字节
   c. DPAPI 解密（CurrentUser 范围）
   d. UTF-8 解码为字符串
   e. JSON 反序列化为 Dictionary<string, string>
   f. account.SetSensitiveTokens(dict) 回填到各字段
```

**安全保证**：
- 只有当前 Windows 用户能解密（DPAPI 与用户 SID 绑定）
- 文件被复制到其他电脑 → 解密失败（用户 SID 不同）
- 用户重装系统 → 解密失败（重新登录即可）

### 3.5 authlib-injector 注入参数生成

**位置**：`YCL.Core\Accounts\AuthlibInjectorHelper.cs`

authlib-injector 通过 Java agent 机制拦截 Minecraft 的登录请求，重定向到第三方认证服务器。

**启动时需要的两个 JVM 参数**：

```
-javaagent:authlib-injector.jar=服务器地址
-Dauthlibinjector.yggdrasil.prefetched=<服务器元数据的 Base64>
```

**`EnsureJarAsync` 流程**：
1. 检查 `%AppData%\YCL\authlib-injector.jar` 是否存在
2. 不存在则从 `https://authlib-injector.yushi.moe/artifact/latest.json` 获取最新版信息
3. 下载 jar 到上述路径
4. 校验 SHA1（如果有）
5. 返回路径

**预取元数据**（`prefetched` 参数）：
- 服务器元数据是一个 JSON，包含认证服务器的公钥、`meta` 等信息
- YCL 启动前先 GET 服务器元数据 → Base64 编码 → 通过 `-Dauthlibinjector.yggdrasil.prefetched` 传给 java
- 这样游戏启动时不用再联网请求元数据，加快启动速度
- 内存缓存：`MetadataCache` 字典缓存服务器地址 → Base64 元数据，避免重复下载

### 3.6 模组 jar 元数据解析

**位置**：`YCL.Core\Mods\LocalModManager.cs`

模组 jar 实际是 zip 压缩包，元数据文件放在 zip 内特定路径。YCL 用 `ZipArchive` 打开 jar，按优先级尝试三种格式：

#### 1. fabric.mod.json（Fabric 模组）

路径：`fabric.mod.json`（zip 根目录）

格式：JSON
```json
{
  "id": "jei",
  "name": "Just Enough Items",
  "version": "4.16.1.302",
  "description": "...",
  "authors": ["mezz"],
  "iconPath": "assets/jei/icon.png"
}
```

#### 2. META-INF/mods.toml（Forge 1.13+）

路径：`META-INF/mods.toml`

格式：TOML
```toml
modLoader="javafml"
loaderVersion="[40,)"
license="MIT"

[[mods]]
modId="jei"
version="4.16.1.302"
displayName="Just Enough Items"
description="..."
```

YCL 用简单字符串处理解析 TOML（不引入新 NuGet 包），只提取 `modId`、`version`、`displayName`、`description`。

#### 3. mcmod.info（Forge 1.12-）

路径：`mcmod.info`

格式：JSON 数组
```json
[{
  "modid": "jei",
  "name": "Just Enough Items",
  "version": "4.16.1.302",
  "description": "..."
}]
```

**图标提取**：从 zip 中读取图标字节流，释放到 `%LocalAppData%\YCL\mod_icons\<hash>.png` 缓存，UI 直接引用。

**容错策略**：单个 jar 解析失败只记日志，不抛异常，仍加入一个最简 `ModInfo`（仅文件名）。

### 3.7 版本隔离实现

**位置**：`YCL.Core\Launch\LaunchArgumentBuilder.cs:50-57`

**原理**：Minecraft 启动参数中 `--gameDir` 指向哪个目录，游戏就把那个目录当工作目录。所有 mods、saves、configs、logs、screenshots 都从工作目录读。

**未隔离**：
```
--gameDir .minecraft/
```
所有版本共享 `.minecraft/mods`、`.minecraft/saves` 等。

**启用隔离**：
```
--gameDir .minecraft/versions/<versionId>/
```
每个版本用自己独立的 `mods`、`saves`、`configs` 等。

**关键代码**：

```csharp
string gameDirectory = minecraftPath;
if (enableVersionIsolation && !string.IsNullOrEmpty(info.Id))
{
    gameDirectory = Path.Combine(minecraftPath, "versions", info.Id);
}
```

**注意**：`assets_root` 与 `library_directory` 仍指向 `.minecraft/`（共享），因为这些文件版本无关、体积大，没必要重复存。

### 3.8 主题系统（iNKORE ThemeManager）

**位置**：`YCL\Services\ThemeService.cs`

YCL 通过 `ThemeManager.Current` 全局单例控制主题，封装在 `ThemeService` 中。

**核心 API**（经反射确认）：

| API | 类型 | 说明 |

| `ThemeManager.Current` | 静态单例 | 全局唯一实例 |
| `ThemeManager.Current.ApplicationTheme` | `ApplicationTheme?` | 设为 `Light`/`Dark` 强制；设为 `null` 跟随系统 |

**枚举映射**：

**应用范围**：iNKORE 设置后立即更新整个应用的资源字典，所有用 `ThemeResource` 标记扩展的控件都会响应，**无需重启**。

**MainWindow 现代化样式**：

```xml
<ui:WindowHelper.UseModernWindowStyle>True</ui:WindowHelper.UseModernWindowStyle>
<ui:WindowHelper.SystemBackdropType>Mica</ui:WindowHelper.SystemBackdropType>
```

`Mica` 是 Windows 11 的半透明背景材质，需要 Windows 11 22000+ 才有效，低版本自动降级。

**配置持久化**：`ThemeMode` 与 `AccentColorHex` 存在 `AppConfig`，启动时 `App.OnStartup` 调用 `themeService.ApplyFromConfig(configService)` 应用。

## 4. 技术选型说明

### 4.1 .NET 10

**理由**：
- 性能：.NET 10 在 JIT、GC、文件 I/O 上比 .NET 8 显著提升
- 长期支持：.NET 10 是 LTS（长期支持）版本，支持到 2028 年
- 新特性：原生 `AOT` 改进、`ZipFile` 性能优化、`System.Text.Json` 增强
- 学习目的：项目本身作为 .NET 10 + WPF 实践项目

**约束**：项目文件 `YCL.csproj` 中明确 `<TargetFramework>net10.0-windows10.0.18362.0</TargetFramework>`，对应 Windows 10 1809（构建 18362）。

### 4.2 WPF + iNKORE.UI.WPF.Modern

**WPF 选择理由**：
- 成熟稳定，文档丰富
- 数据绑定强大（`INotifyPropertyChanged` + `ObservableCollection`）
- XAML 分离视图与逻辑，设计时预览
- 适合桌面复杂界面（启动器需要大量列表、表格、对话框）

**iNKORE.UI.WPF.Modern 选择理由**：
- Windows 11 风格控件（`NavigationView`、`NumberBox`、`InfoBar` 等）
- 内置 `ThemeManager`，亮色/暗色/强调色一行切换
- Mica 半透明背景（Windows 11）
- 开源（MIT 协议），活跃维护
- 对标 Windows App SDK 控件库，但能在 .NET 桌面直接用

### 4.3 CommunityToolkit.Mvvm

**版本**：8.4.2

**理由**：
- 官方维护，Microsoft 长期支持
- 源生成器（Source Generator）自动生成 `INotifyPropertyChanged` 样板
- `[ObservableProperty]` 特性 → 自动生成公开属性
- `[RelayCommand]` 特性 → 自动生成 `ICommand` 实现
- `ObservableObject` 基类提供 `SetProperty<T>(ref field, value)`
- 无运行时反射，性能好

**示例**：

```csharp
public partial class HomePageViewModel : ViewModelBase
{
    [ObservableProperty]
    private string _searchText = string.Empty;

    [RelayCommand]
    private async Task SearchAsync()
    {
        // 自动生成 SearchCommand 属性供 XAML 绑定
    }
}
```

### 4.4 Microsoft.Extensions.DependencyInjection

**版本**：10.0.10

**理由**：
- .NET 官方 DI 容器，与 `Microsoft.Extensions.Hosting` 无缝集成
- 支持构造函数注入、工厂模式注册、集合注入（`IEnumerable<T>`）
- 三种生命周期：Singleton / Scoped / Transient（YCL 用 Singleton 与 Transient）
- 类型安全，编译期检查依赖
- 生态丰富（日志、配置、选项模式等）

### 4.5 System.Text.Json

**理由**：
- .NET 内置，无需引入 Newtonsoft.Json
- 性能优于 Newtonsoft（基于 `Span<T>` 与 UTF-8 直接读写）
- 内存分配少，适合高频序列化场景
- 支持多态序列化（`[JsonPolymorphic]` + `[JsonDerivedType]`），YCL 用它处理 `AccountBase` 子类
- 异步 API（`JsonSerializer.SerializeAsync`）

**配置示例**：

```csharp

```

**自定义 Converter**：`VersionResolver` 中注册了 `ArgumentItemConverter` 处理 `arguments.game` 数组中混合 `string` 与对象的情况。

### 5.6 日志位置与级别

**位置**：`%AppData%\YCL\logs\log_YYYY-MM-DD.txt`

**文件名示例**：`log_2026-07-17.txt`

**级别**（从低到高）：

| 级别 | 用途 | 示例 |
|------|------|------|
| `DEBUG` | 详细调试信息（默认不输出，开发时用） | 启动参数详情、HTTP 请求 URL |
| `INFO` | 正常流程节点 | "版本解析完成"、"账户已加载" |
| `WARN` | 非致命异常（自动恢复） | "单个模组解析失败"、"配置文件不存在，用默认" |
| `ERROR` | 错误（可能影响功能） | "启动游戏失败"、"微软登录失败" |

**格式**：

```
[HH:mm:ss.fff] [T<threadId>] [LEVEL] <message>
  → 异常类型: <Type>
  → 异常消息: <Message>
  → 堆栈跟踪:
<StackTrace>
```

**调用方式**：

```csharp
using YCL.Core.Utils;

Logger.Info("版本解析完成");
Logger.Warn($"扫描模组失败：{file} - {ex.Message}");
Logger.Error("启动游戏失败", ex);  // ex 为 Exception 对象
Logger.Debug("完整命令行：" + cmdLine);
```

**线程安全**：内部用 `lock (_lock)` 保护，多线程并发写不会错乱。

**滚动策略**：按日期分文件，每天一个新文件。**没有自动清理**，需要用户手动管理（避免占满磁盘）。

## 6. 构建与发布

### 6.1 开发环境要求

| 工具 | 版本 | 用途 |
|------|------|------|
| .NET SDK | 10.0 或更高 | 编译、运行 |
| Visual Studio 2022 | 17.10+ | 推荐 IDE，完整 WPF 设计器 |
| VS Code + C# Dev Kit | 最新 | 轻量替代，无 XAML 设计器 |
| Inno Setup | 6.0+ | 制作安装包（仅发布时需要） |
| Git | 任意 | 版本控制 |

**验证环境**：

```powershell
dotnet --version
# 应输出 10.0.x 或更高
```

### 6.2 构建命令

#### 调试构建

```powershell
# 在解决方案根目录 e:\Your Craft Launcher 下
dotnet build YCL.sln -c Debug
```

输出：`YCL\bin\Debug\net10.0-windows10.0.18362.0\YCL.exe`

#### Release 构建

```powershell
dotnet build YCL.sln -c Release
```

输出：`YCL\bin\Release\net10.0-windows10.0.18362.0\YCL.exe`

#### 发布（框架依赖）

```powershell
dotnet publish YCL\YCL.csproj -c Release -o publish
```

输出到 `publish\` 目录。这种方式**不包含 .NET 运行时**，用户电脑需要装 .NET 10 桌面运行时。

#### 发布（自包含）

```powershell
dotnet publish YCL\YCL.csproj -c Release -o publish-selfcontained -r win-x64 --self-contained true
```

输出包含 .NET 运行时，体积大（约 80MB），但用户**无需另装 .NET**。

**YCL 默认采用框架依赖**（与 PCL-CE 一致），减小体积。

### 6.3 安装包制作（Inno Setup）

**前置条件**：
1. 已安装 Inno Setup 6（下载：https://jrsoftware.org/isdl.php）
2. 已执行 `dotnet publish` 生成 `publish\` 目录

**编译命令**（在项目根目录）：

```powershell
"C:\Program Files (x86)\Inno Setup 6\ISCC.exe" installer\YCL.iss
```

或用图形界面：

1. 双击 `installer\YCL.iss` 用 Inno Setup Compiler 打开
2. 按 `Ctrl+F7` 或菜单 `Build → Compile`

**输出**：`installer\output\YCL-Setup-1.0.0.exe`

**YCL.iss 关键配置**（`installer\YCL.iss`）：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `AppName` | Your Craft Launcher | 显示名 |
| `AppVersion` | 1.0.0 | 版本号 |
| `DefaultDirName` | `{autopf}\YCL` | 默认安装目录（自动 32/64 位） |
| `Compression` | lzma2 | 高压缩率 |
| `SolidCompression` | yes | 固实压缩，体积更小 |
| `ArchitecturesAllowed` | x64compatible | 仅 64 位 |

**多语言支持**：
- 简体中文（默认）：`ChineseSimplified.isl`
- 英文：`Default.isl`

### 6.4 发布注意事项

#### 框架依赖 vs 自包含

| 类型 | 体积 | 用户需装 .NET | 适用场景 |
|------|------|----------------|----------|
| 框架依赖（默认） | ~15MB | 是 | 桌面应用常规发布，与 PCL-CE 一致 |
| 自包含 | ~80MB | 否 | 离线环境、企业内网 |

**YCL 选框架依赖**，原因：
- 体积小，下载快
- .NET 10 桌面运行时将来会预装在 Windows 中（类似 .NET Framework）
- 安装包首次运行会引导用户下载运行时

#### 版本号管理

`YCL.csproj` 中三个版本号（**修改时三个要一起改**）：

```xml
<AssemblyVersion>1.0.0.0</AssemblyVersion>   <!-- CLR 用，四段式 -->
<FileVersion>1.0.0.0</FileVersion>          <!-- 文件属性用，四段式 -->
<Version>1.0.0</Version>                     <!-- 显示用，三段式 -->
```

同步更新位置：
- `YCL\YCL.csproj`
- `installer\YCL.iss` 中的 `AppVersion`
- `YCL.Core\Launch\LaunchArgumentBuilder.cs` 中的 `LauncherVersion` 常量
- `YCL.Core\Update\UpdateChecker.cs` 中的版本比较逻辑（自动从 Assembly 读取，无需手改）
- GitHub Release 的 tag 名（如 `v1.0.1`）

#### 应用图标

当前项目未提供 `.ico` 文件。添加方式：

1. 准备 `YCL.ico`（多尺寸：16/32/48/256）
2. 放到 `YCL\` 目录
3. 在 `YCL.csproj` 取消注释：
   ```xml
   <ApplicationIcon>YCL.ico</ApplicationIcon>
   ```
4. `installer\YCL.iss` 中的 `UninstallDisplayIcon` 会自动从 `YCL.exe` 提取图标
5. 重新 `dotnet publish`

#### 发布前自检清单

发布前确认：
- [ ] 版本号三处一致（`csproj` / `iss` / `LaunchArgumentBuilder`）
- [ ] `publish\` 目录已重新生成（`dotnet publish`）
- [ ] 安装包编译成功（`ISCC.exe installer\YCL.iss`）
- [ ] 在干净环境（没装 .NET）测试安装包能正常安装并启动
- [ ] GitHub Release 已创建，tag 名符合 `vX.Y.Z` 格式
- [ ] Release notes 描述本次更新内容
- [ ] 用户手册与开发文档已同步更新（如有功能变更）

#### CI/CD 建议（可选）

如需自动化，可在项目根目录加 GitHub Actions workflow：

```yaml
# .github\workflows\build.yml
name: Build
on:
  push:
    tags: ['v*']
jobs:
  build:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.0.x'
      - run: dotnet publish YCL\YCL.csproj -c Release -o publish
      - name: Upload artifact
        uses: actions/upload-artifact@v4
        with:
          name: YCL-publish
          path: publish\
```

Inno Setup 编译也可用 GitHub Action（`runner` 用 windows-latest），但需要先装 Inno Setup。
